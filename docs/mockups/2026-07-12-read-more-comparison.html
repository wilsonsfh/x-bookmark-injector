Why B wins

CriterionA — Hard char cutB — CSS clamp ★C — No expansion
Cut point is natural No — char count bisects words, breaks meaning mid-sentence Yes — clamp ends on a rendered line boundary, never mid-word N/A — full text always shown
Feed scroll length Short — ~3 lines visible Controlled — max 6 lines collapsed Variable — long posts push footer far off-screen
Implementation complexity JS substring + char count needed CSS clamp + measured overflow; tiny accessible toggle, no text slicing None — render as-is
Inline "Read more" placement Appended after ellipsis — visually awkward inline splice Follows the clamp fade naturally; feels like X's own "Show more" N/A — no affordance to signal there's more on X
Open on X ↗ retained Yes Yes — footer always visible Yes — sole read-more path
Keep / Done · Remove retained Yes Yes Yes
Works with short posts (<6 lines) Yes (no cut triggered) Yes — clamp is a no-op; "Read more" never renders Yes
Matches X's own "Show more" UX No — char cut is not how X works Yes — line-based clamp is identical to X's native behaviour No — X collapses long posts; always-expanded breaks the pattern

B wins on every axis that matters for a feed-injected card. CSS -webkit-line-clamp:6 is zero-JS, handles any language/RTL, and produces a clean line-boundary cut identical to X's own "Show more" — so the interaction feels native rather than bolted-on. A's hard char cut requires JS, bisects words, and looks broken on non-ASCII text. C avoids the problem but trades it for unbounded card height, which disrupts feed rhythm and removes the incentive to open the post on X (the whole point of the tool is the Done/Keep loop, not full in-feed reading). The footer — Open on X ↗ · Keep · Done · Remove — is identical across all three; only the text body treatment changes.
